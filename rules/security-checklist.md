# Company Security Checklist v2

## Authentication & access control

- [ ] Enforce MFA/2FA for all users, admins, and service accounts [CRITICAL]
- [ ] Implement strong password policies — min length, complexity, history checks [HIGH]
- [ ] Integrate leaked-password lookup (e.g. Have I Been Pwned API) [HIGH]
- [ ] Enforce session timeouts with automatic logout on inactivity [HIGH]
- [ ] Invalidate all sessions globally on password reset [CRITICAL]
- [ ] Use HttpOnly, SameSite=Strict, and Secure flags on all session cookies [CRITICAL]
- [ ] Validate RBAC to prevent horizontal and vertical privilege escalation [CRITICAL]
- [ ] Enforce tenant isolation — cross-tenant data leakage must be impossible [CRITICAL]
- [ ] CAPTCHA or proof-of-work on login, registration, and password-reset pages [HIGH]
- [ ] Limit concurrent active sessions per user account [MEDIUM]
- [ ] Implement account lockout after N failed login attempts with exponential backoff [CRITICAL]
- [ ] Use time-limited, single-use tokens for email verification and password reset links [CRITICAL]
- [ ] Log all authentication events — successes, failures, and MFA bypass attempts [HIGH]
- [ ] Enforce IP allowlisting or step-up auth for admin dashboard access [HIGH]
- [ ] Rotate all API keys and service credentials on a defined schedule [HIGH]

## API & application security

- [ ] Sanitize and validate all user inputs server-side — whitelist approach [CRITICAL]
- [ ] Encode all user-generated output contextually to prevent XSS [CRITICAL]
- [ ] Use parameterized queries or ORM exclusively — no raw SQL with user input [CRITICAL]
- [ ] Protect all state-changing requests against CSRF [CRITICAL]
- [ ] Enforce rate limiting and throttling on all public API endpoints [CRITICAL]
- [ ] Use strong API auth — OAuth 2.0, OpenID Connect, or signed JWTs [CRITICAL]
- [ ] Validate JWT signatures, expiration, and issuers on every request [CRITICAL]
- [ ] Disable unnecessary HTTP methods (TRACE, OPTIONS where not needed) [HIGH]
- [ ] Disable detailed error messages and stack traces in production [HIGH]
- [ ] Scan and update all open-source dependencies (Snyk, Dependabot) [HIGH]
- [ ] Implement Content Security Policy (CSP) headers on all pages [HIGH]
- [ ] Set X-Frame-Options, X-Content-Type-Options, and Referrer-Policy headers [HIGH]
- [ ] Validate file uploads — type, size, MIME, and scan for malware before storing [CRITICAL]
- [ ] Strip metadata (EXIF, document properties) from uploaded files before storage [MEDIUM]
- [ ] Enforce CORS policy — whitelist allowed origins explicitly [HIGH]
- [ ] Use SSRF protection — block internal IP ranges from outbound requests triggered by user input [CRITICAL]
- [ ] Prevent path traversal in file operations — never concatenate user input into file paths [CRITICAL]
- [ ] Ensure all redirects are validated against an allowlist — prevent open redirect attacks [HIGH]
- [ ] Implement sub-resource integrity (SRI) for all third-party CDN scripts [MEDIUM]
- [ ] Audit all third-party webhooks — verify signatures before processing payload [CRITICAL]

## Data protection & privacy

- [ ] Enforce TLS 1.3 (or 1.2 minimum) for all data in transit [CRITICAL]
- [ ] Implement HSTS headers with preloading [HIGH]
- [ ] Encrypt sensitive data at rest using AES-256 or equivalent [CRITICAL]
- [ ] Manage encryption keys separately from encrypted data volumes [CRITICAL]
- [ ] Hash passwords using Argon2id or bcrypt with per-user salt [CRITICAL]
- [ ] Mask or tokenize sensitive data (credit cards, PII) in logs and UI [CRITICAL]
- [ ] Automated, encrypted backups — test restoration on a schedule [HIGH]
- [ ] Publish a compliant Privacy Policy and Terms of Service [HIGH]
- [ ] Document and test a Right to be Forgotten / data deletion workflow [HIGH]
- [ ] Verify compliance requirements — GDPR, CCPA, HIPAA, PCI-DSS as applicable [HIGH]
- [ ] Classify all data you store — PII, sensitive, public — and apply handling rules per class [CRITICAL]
- [ ] Enforce data minimization — only collect data you actually need [HIGH]
- [ ] Implement retention policies — auto-delete or anonymize data after defined period [HIGH]
- [ ] Encrypt all inter-service communication inside the network (mTLS) [CRITICAL]
- [ ] Prevent secret leakage in API responses — never return raw tokens, keys, or hashes to clients [CRITICAL]
- [ ] Ensure database connection strings and secrets are never logged [CRITICAL]
- [ ] Deploy Data Loss Prevention (DLP) to detect and block unauthorized data transfers [HIGH NEW]
- [ ] Enforce vector/embedding namespace isolation per tenant — embeddings from one tenant must never surface in another tenant's retrieval results [CRITICAL NEW]

## Cloud infrastructure & network security

- [ ] Close all unused network ports and disable unused services in production [CRITICAL]
- [ ] Restrict database access to application tier only — no public IPs [CRITICAL]
- [ ] Ensure all cloud storage buckets are private by default [CRITICAL]
- [ ] Remove all hardcoded credentials and API keys from repositories [CRITICAL]
- [ ] Inject secrets at runtime via a dedicated Secret Manager [CRITICAL]
- [ ] Deploy a Web Application Firewall (WAF) [HIGH]
- [ ] Enable DDoS protection at DNS or CDN layer [HIGH]
- [ ] Isolate dev, staging, and production networks completely [CRITICAL]
- [ ] Restrict SSH/RDP access behind a VPN or Bastion host [CRITICAL]
- [ ] Enable automated security patching for server OSes [HIGH]
- [ ] Scan all container images for known CVEs before deployment [CRITICAL]
- [ ] Enforce least-privilege IAM — every service and user has only the permissions it needs [CRITICAL]
- [ ] Enable cloud provider audit logging (CloudTrail, GCP Audit Logs, Azure Monitor) [CRITICAL]
- [ ] Use immutable infrastructure — no manual changes to production servers [HIGH]
- [ ] Enforce network segmentation — separate VPCs or subnets per environment [HIGH]
- [ ] Tag all cloud resources with owner and environment [MEDIUM]
- [ ] Review and revoke unused cloud credentials and service accounts quarterly [HIGH]
- [ ] Disable root/admin cloud account for daily use — use IAM roles instead [CRITICAL]

## CI/CD & supply chain security

- [ ] Run SAST tools inside CI/CD pipeline on every pull request [CRITICAL]
- [ ] Pin all dependency versions — no floating ranges in package.json or requirements.txt [HIGH]
- [ ] Enable automated dependency vulnerability alerts (Dependabot, Snyk) [HIGH]
- [ ] Require signed commits for all contributors [HIGH]
- [ ] Enforce branch protection — require PR reviews and status checks before merge [HIGH]
- [ ] Scan git history for accidentally committed secrets (git-secrets, TruffleHog) [CRITICAL]
- [ ] Use a dedicated build environment — never build production artifacts on developer machines [HIGH]
- [ ] Store build artifacts in a signed, tamper-evident registry [MEDIUM]
- [ ] Audit all GitHub Actions or CI workflows for third-party action risks [HIGH]
- [ ] Restrict production deployment permissions to CI system only — no manual pushes [CRITICAL]

## AI & LLM security (platform-specific)

- [ ] Implement multi-layer prompt injection defense — validate and sanitize all user inputs before they reach the LLM context [CRITICAL NEW]
- [ ] Validate and sanitize all LLM outputs before returning to users — scan for PII, secrets, and system prompt leakage [CRITICAL NEW]
- [ ] Never include tenant secrets, API keys, or PII in LLM system prompts [CRITICAL NEW]
- [ ] Enforce strict tenant namespace isolation in vector/embedding databases — row-level security at the embedding layer [CRITICAL NEW]
- [ ] Monitor Time-To-First-Token patterns across tenants — KV cache timing can leak cross-tenant inference activity [HIGH NEW]
- [ ] Enforce authorization bounds on all LLM tool calls — never delegate access control decisions to the LLM itself [CRITICAL NEW]
- [ ] Enforce schema validation and parameter guards on every state-mutating LLM tool call (add to cart, initiate return, etc.) [CRITICAL NEW]
- [ ] Rate-limit LLM API queries per tenant — prevent model extraction attacks via systematic API querying [HIGH NEW]
- [ ] Log all LLM inputs and outputs for audit — store separately from general application logs with restricted access [CRITICAL NEW]
- [ ] Implement output filtering to prevent the LLM from generating content that reveals other tenants' data, personas, or configurations [CRITICAL NEW]
- [ ] Validate that safety engine overrides are structurally impossible — LLM must not be able to instruct the system to skip SafetyEngine [CRITICAL NEW]
- [ ] Never pass raw user messages directly into LLM system prompt position — only into user turn position [CRITICAL NEW]
- [ ] Maintain a Software Bill of Materials (SBOM) for all AI/ML dependencies and model versions in use [HIGH NEW]
- [ ] Monitor LLM provider (Groq, Gemini) security advisories — subscribe to breach notifications [HIGH NEW]
- [ ] Test all persona configurations to verify they cannot be used to bypass safety rules or extract system prompts [CRITICAL NEW]

## Monitoring, logging & incident response

- [ ] Centralize logs to a secure external logging server [CRITICAL]
- [ ] Never log sensitive data — passwords, credit cards, PII, tokens [CRITICAL]
- [ ] Real-time alerts for brute-force, anomalous access, and system errors [CRITICAL]
- [ ] Configure immutable log storage to prevent tampering [HIGH]
- [ ] Draft an Incident Response Plan covering detection, containment, and public disclosure [CRITICAL]
- [ ] Create a security@yourdomain.com alias for vulnerability reports [HIGH]
- [ ] Publish a security.txt in the website root directory [MEDIUM]
- [ ] Monitor for anomalous API usage — unusual volume, new IPs, off-hours access [HIGH]
- [ ] Set up uptime and latency alerting [HIGH]
- [ ] Conduct tabletop incident response drills at least twice per year [HIGH]
- [ ] Define and document RTO and RPO for each critical service [HIGH]
- [ ] Test disaster recovery procedures — restore from backup in a clean environment [CRITICAL]
- [ ] Maintain a public status page — communicate incidents transparently [MEDIUM]
- [ ] Comprehensive audit log: every data access, modification, deletion, and admin action must be logged with who, what, when, where [CRITICAL NEW]
- [ ] Set up anomaly detection for LLM usage — alert if a single tenant's LLM consumption spikes abnormally [HIGH NEW]

## Vulnerability testing & assurance

- [ ] Run SAST tools in CI/CD pipeline [CRITICAL]
- [ ] Execute DAST scans (OWASP ZAP or Burp Suite) against staging environment [HIGH]
- [ ] Schedule an external third-party penetration test before public launch [CRITICAL]
- [ ] Resolve or formally accept all findings from scans and pen tests [CRITICAL]
- [ ] Test all tenant isolation boundaries — data from tenant A must never reach tenant B [CRITICAL]
- [ ] Test all authentication flows — brute force, session fixation, token reuse [CRITICAL]
- [ ] Conduct fuzzing on all public API input fields [HIGH]
- [ ] Verify all error states return safe, generic messages — no stack traces or schema hints [HIGH]
- [ ] Run a dependency audit (npm audit / pip-audit) before every release [HIGH]
- [ ] Include security regression tests in the CI test suite [HIGH]
- [ ] Test all LLM prompt injection vectors — attempt to extract system prompt, bypass safety, and access other tenant data via crafted prompts [CRITICAL NEW]
- [ ] Conduct a business logic abuse test — attempt to use the AI to execute unauthorized commerce actions [CRITICAL NEW]

- [ ] Security awareness training for all employees and contractors — minimum annual [CRITICAL]
- [ ] Enforce a clean desk and screen-lock policy [MEDIUM]
- [ ] Run phishing simulation exercises quarterly [HIGH]
- [ ] Define and enforce a secure development lifecycle (SDL) for every project [CRITICAL]
- [ ] Conduct background checks for employees with production access [HIGH]
- [ ] Maintain an asset inventory — every device with production access is registered and managed [HIGH]
- [ ] Enforce MDM on all devices accessing production systems [HIGH]
- [ ] Offboarding checklist — immediately revoke all access and credentials when someone leaves [CRITICAL]
- [ ] Require a signed confidentiality agreement from all team members and contractors [HIGH]

## Third-party & vendor risk

- [ ] Audit every third-party service for data handling and compliance (SOC 2, ISO 27001) [HIGH]
- [ ] Ensure vendor contracts include a Data Processing Agreement (DPA) where required by GDPR [HIGH]
- [ ] Review permissions granted to every OAuth integration — revoke excessive scopes [CRITICAL]
- [ ] Maintain an inventory of all third-party SDKs and services in production [HIGH]
- [ ] Monitor for third-party service breach notifications — subscribe to vendor security bulletins [HIGH]
- [ ] Do not embed third-party analytics or tracking scripts without a CSP and SRI hash [HIGH]
- [ ] Review and document the data handling policies of every LLM provider you send user data to (Groq, Google, OpenAI) [CRITICAL NEW]
- [ ] Confirm LLM providers do not train on your API request data — verify opt-out status in their terms [CRITICAL NEW]

## Compliance & certification roadmap

- [ ] Map all data types and flows to applicable regulations — GDPR, CCPA, PCI-DSS, HIPAA [CRITICAL NEW]
- [ ] Document your Security Policy, Acceptable Use Policy, Change Management Policy, and Business Continuity Plan [CRITICAL NEW]
- [ ] Begin SOC 2 Type II preparation at least 6 months before you need it for enterprise sales [HIGH NEW]
- [ ] Implement a formal risk assessment process — identify, score, and document risks annually [HIGH NEW]
- [ ] Establish a change management process — all infrastructure and code changes are reviewed and approved [HIGH NEW]
- [ ] Maintain an evidence library for all security controls — auditors require proof of implementation, not just claims [HIGH NEW]
- [ ] Implement a formal patch management schedule — define SLAs for critical (24h), high (7d), medium (30d) vulnerabilities [CRITICAL NEW]
- [ ] Publish a breach notification process — document timelines for notifying regulators and users (GDPR requires 72 hours) [CRITICAL NEW]
- [ ] Conduct a formal annual security review — reassess all controls, threats, and risk posture [HIGH NEW]
- [ ] Define and publish an internal Acceptable Use Policy for all AI tools used in development [MEDIUM NEW]
